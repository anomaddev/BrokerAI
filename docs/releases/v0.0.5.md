# BrokerAI v0.0.5 — Alpha

**Release date:** 2026-07-01  
**Tag:** `v0.0.5`  
**Commit:** `02a1247`

Alpha 0.0.5 is a major architectural and trading release. BrokerAI moves from research-and-visibility (0.0.3) to a **Secretary-coordinated trading loop** with strategy-driven forex analysis, OANDA order execution, a MongoDB trade ledger, and a full trading workspace in the web UI.

> **Note:** Live order execution is **forex-only via OANDA** in this release. Use a **practice** account while evaluating. Stocks, crypto, futures, options, and precious metals remain scaffolded. Backtesting is a placeholder page.

---

## Highlights

- **The Loop** — Secretary-coordinated pipeline with spin-up/spin-down workers for data fetch, analysis, and execution
- **Live forex execution** — EMA Crossover and Custom strategies can place OANDA market orders when gates pass
- **Strategy builder** — parameterized presets (timeframe, signal, filters, risk, execution) saved to MongoDB
- **Trade ledger** — open/closed trades with reason codes, P&L, and broker reconciliation
- **OANDA trade sync** — import broker trades, close drift, backfill exit details; manual sync from the UI
- **Explore** — interactive candle charts with live revision stream and recent-pair history
- **Strategy Analysis** — browse analyzer runs with direction, confidence, filter summary, and execution outcome
- **Market sessions bar** — live forex session and asset-class status in the app chrome
- **Bot activity timeline** — correlated pipeline events with job IDs
- **Background tasks** — long-running sync and research jobs with progress in the UI

## Architecture — The Loop

v0.0.5 replaces the legacy five independent tick bots with a Secretary-coordinated model. Persistent bots manage scheduling and authority; ephemeral workers handle per-symbol work.

| Persistent | Role |
|------------|------|
| **Orchestrator** | Process lifecycle, heartbeat, control IPC |
| **Secretary** | Candle timeline, pipeline dispatch, timed research, activity log |
| **Broker** | Execution gates, exit monitors, associate dispatch, periodic OANDA sync |

| Ephemeral worker | Role |
|------------------|------|
| **Data Manager** | Fetch/cache candles per `(pair, timeframe)` request |
| **Data Analyst** | Run strategy analysis per WorkUnit |
| **Associate** | Place orders per asset class (forex implemented) |
| **Researcher** | Daily/weekly reports on demand |

### M15 pipeline flow

At each due candle close:

1. Secretary creates a `CandleJob` + `PipelineContext`
2. Data Manager worker upserts bars into `market_data`
3. Data Analyst worker evaluates enabled strategies
4. Broker applies execution gates, resolves priority conflicts, dispatches Associates
5. Broker exit monitors evaluate open trades for crossover / trail stops

Parallelism is capped by `BROKERAI_PIPELINE_CONCURRENCY` (default **10**).

### Configuration

```bash
BROKERAI_ENABLED_BOTS=secretary,broker,researcher
BROKERAI_USE_SECRETARY_PIPELINE=true
BROKERAI_PIPELINE_CONCURRENCY=10
BROKERAI_SECRETARY_TICK_INTERVAL_SECONDS=5
BROKERAI_BROKER_SYNC_INTERVAL_SECONDS=30
```

**Legacy rollback** (pre-0.0.5 tick bots):

```bash
BROKERAI_USE_SECRETARY_PIPELINE=false
BROKERAI_ENABLED_BOTS=data_manager,data_analyzer,executor,brokers,researcher
```

See [`docs/architecture/the-loop.md`](../../docs/architecture/the-loop.md) for full design notes.

## Trading

### Strategy presets

| Preset | Status | Description |
|--------|--------|-------------|
| **EMA Crossover** | Implemented | Fast/slow EMA cross with ADX + ATR filters, ATR-based stops, RR take-profit |
| **Custom** | Implemented | Blank template for manual parameter configuration |

Params use **schema v1** with sections: `timeframe`, `indicators`, `signal`, `filters`, `exits`, `risk`, `execution`. See [`docs/strategies/params-schema.md`](../../docs/strategies/params-schema.md).

Default EMA Crossover behavior:

- **Timeframe:** M15
- **Signal:** bullish/bearish EMA crossover on bar close
- **Filters:** ADX ≥ 25, ATR minimum
- **Sessions:** London + NY
- **Risk:** 1% per trade, max 3 trades/day per strategy/pair

### Execution gates

Before placing an order, Broker checks:

- Valid signal direction
- Confidence ≥ `execution.min_confidence`
- Asset-class session active (from broker settings)
- Strategy session active (`execution.sessions`)
- Daily trade count below `risk.max_trades_per_day`
- Priority resolution when multiple strategies signal the same pair

### Exit monitors

Open trades are monitored on each relevant candle close:

- **Reverse crossover** — opposite EMA cross
- **Trail stop (EMA slow)** — price crosses trailing slow EMA
- **Trail stop (ATR)** — ATR-based trailing stop

Manual closes and broker-side closes are also recorded.

### Forex associate (OANDA)

When OANDA is configured, the forex associate:

1. Validates asset-class and session gates
2. Sizes units from risk % and stop distance (or uses explicit units)
3. Places a market order with optional stop-loss / take-profit
4. Records the trade in the ledger with `broker_order_id`

Without OANDA credentials, intents are logged to the ledger only (paper mode).

### Indicators

Built-in indicators used by strategies and exit monitors: **EMA**, **SMA**, **RSI**, **ADX**, **ATR**.

## Trade ledger

Trades are stored in MongoDB (`trades` collection) with:

- Strategy linkage (`strategy_id`, `strategy_name`)
- Direction, pair, units, entry/exit prices
- Open/closed status with timestamps
- **Reason codes** — machine codes resolved to human labels (e.g. `bullish_cross` → "Bullish crossover", `broker_closed` → "Closed on OANDA")
- Close metadata — exit price, realized P&L, broker timestamps

### OANDA sync

`sync_oanda_trades_to_ledger()` reconciles the ledger against live OANDA state:

| Action | Description |
|--------|-------------|
| **Import** | Broker-open trades not in the ledger → `oanda_import` |
| **Update** | Matched open trades — refresh units, prices, unrealized P&L |
| **Close** | Ledger-open trades gone from broker → `broker_closed` with exit details |
| **Backfill** | Closed ledger trades missing P&L → fetch from OANDA trade history |

Sync runs automatically on the Broker slow tick (~30s) and manually via **Trading → Trades → Sync OANDA**.

### Reconciliation API

`GET /api/trades/reconciliation` compares ledger open trades vs OANDA snapshot and returns badge hints (`matched`, `ledger_only`, `broker_only`).

### Trade reason registry

Reason codes are categorized as **signal**, **exit**, **import**, **manual**, **broker**, or **other**. The Trades table shows compact short labels (≤19 chars) with full labels in tooltips.

## Upgrade from v0.0.3

1. Set update track to `release` with pin `0.0.5`, or pull `main`
2. Apply via **Settings → System → Updates** or `brokerai update check`
3. Update config for Secretary mode:

   ```bash
   BROKERAI_ENABLED_BOTS=secretary,broker,researcher
   BROKERAI_USE_SECRETARY_PIPELINE=true
   ```

4. Configure:
   - OANDA in **Settings → Exchange Connections** (practice recommended)
   - Enabled forex pairs in **Settings → Broker → Forex**
   - At least one **enabled strategy** on overlapping pairs in **Trading → Strategies**
5. Start the orchestrator: `brokerai run orchestrator` or systemd `brokerai-orchestrator.service`
6. Optional: Massive API key for extended market-status indicators in the sessions bar

No database migration script is required — new collections are created on first access.

## Known limitations (Alpha)

- **Forex execution only** — other asset classes have associate/data-manager stubs (`TODO(loop)`)
- **Practice account strongly recommended** — live OANDA execution is real money
- **Fixed account balance for sizing** — forex associate uses a 10,000 default when OANDA balance is unavailable in the sizing path
- **Account snapshots in-memory** — not yet persisted to MongoDB
- **Backtesting not implemented** — placeholder page only
- **Researcher trade-analysis mode** — not implemented
- **Legacy CLI bots** — `brokerai run data-manager` etc. still work when `BROKERAI_USE_SECRETARY_PIPELINE=false`, but Secretary mode is the supported path
- **Pre-existing import defect** — `brokerai.cli.candles` imports a missing `brokerai.trading.data.candle_cache` module; this breaks some CLI paths and 6 test files unrelated to the web API (see `AGENTS.md`)
