# BrokerAI v0.0.7 — Alpha

> **Historical note:** Point-in-time alpha notes. Current stack uses Postgres/Supabase and persistent bots `secretary`, `broker`, `researcher`. See [`docs/README.md`](../README.md).

**Release date:** 2026-07-04  
**Tag:** `v0.0.7`  
**Commit:** _(set at release)_

Alpha 0.0.7 is a **broker data-model and OANDA sync** release. It replaces the legacy `trades` collection and dual OANDA polling loops with a unified broker ledger (`broker_lots`, `broker_events`, `broker_sync_state`), a documented three-phase OANDA entity-linkage architecture, richer trade-detail UI, and production-oriented event persistence (bulk upsert, pagination, tiered retention, streaming bootstrap).

> **Note:** Live order execution remains **forex-only via OANDA**. This release focuses on how broker trades, orders, and transactions are normalized, linked, synced, and displayed — not on new strategy types or asset classes.

**Architecture reference:** [`docs/architecture/oanda-entity-linkages.md`](../architecture/oanda-entity-linkages.md)

---

## Highlights

- **Unified OANDA sync** — single orchestrator (`run_broker_sync`); one `/changes` poll per incremental tick; canonical bootstrap sequence
- **Broker ledger** — `PositionLot` + `BrokerEvent` model with embedded child orders (SL/TP), close-reason inference, and full transaction audit trail
- **Child-order state machine** — event-driven SL/TP lifecycle reconciled against trade snapshots and `/changes` order deltas
- **Materialized exposure** — `instrument_exposure` rollups with API + dashboard panel; periodic positions health check
- **Streaming bootstrap** — transaction backfill upserts one OANDA page at a time (memory-bounded for large accounts)
- **Event retention** — trade-linked events kept indefinitely; low-value admin types TTL after 90 days
- **Trade detail workspace** — candle chart, event flags, child-order txn IDs, and close-reason registry entries
- **OANDA HTTP client** — shared `OandaHttpClient` with rate limiting and connection reuse

## Pull request summary

Use this section as the PR description body.

### What changed

BrokerAI’s OANDA integration moves from ad-hoc trade sync + parallel web loops to a single, documented sync spine. Every OANDA trade becomes a `PositionLot`; every transaction becomes a `BrokerEvent`. Child stop-loss and take-profit orders are embedded on lots and kept in sync via an event-driven FSM. Bootstrap and cursor repair fetch **all** transaction pages (not capped at 500). Production bootstrap streams pages to MongoDB without holding full history in memory.

### Why

- Eliminate lock contention and inconsistent bootstrap between `run_broker_sync` and `run_oanda_account_sync`
- Fix correctness gaps (pagination, double upsert on gap repair, `closed_at`/`close_time` query mismatch)
- Enable semantic linkage (SL/TP fill/cancel txn IDs, partial closes, exposure rollups) for trade analysis UI
- Prepare multi-broker abstractions without shipping a second adapter

### Test plan

- [ ] `./venv/bin/python -m pytest` — full suite (258+ tests)
- [ ] Fresh account bootstrap: verify `broker_lots`, `broker_events`, and `broker_sync_state` populated; cursor matches OANDA `lastTransactionID`
- [ ] Incremental sync: single `/changes` call per tick; gap repair events persisted once
- [ ] Open trade with SL/TP: trade detail shows child order txn IDs; close reason `stop_loss` / `take_profit` when applicable
- [ ] Exchange dashboard: instrument exposure panel matches open lots
- [ ] Account with >500 historical transactions: bootstrap completes without OOM; event count matches OANDA

## Architecture — OANDA entity linkages

Three implementation phases (all shipped in 0.0.7):

| Phase | Focus |
|-------|--------|
| **1 — Foundation** | Unified orchestrator, canonical bootstrap, throttled exposure check, stale cursor repair, collapsed web polling loops |
| **2 — Semantics** | Child-order FSM, order-level `/changes` ingestion, materialized `instrument_exposure`, exposure API + UI |
| **3 — Scale & prep** | Bulk event upsert, transaction pagination, streaming bootstrap, tiered retention, multi-broker ID/connection helpers |

### Sync flow (after 0.0.7)

```
run_broker_sync
  ├── bootstrap (first run / full mode)
  │     ├── GET /accounts/{id}     → open lots + lastTransactionID
  │     ├── GET /trades?state=CLOSED
  │     └── iter_transactions_idrange → event_sink (page-by-page upsert)
  ├── incremental (every tick)
  │     ├── GET /changes?sinceTransactionID=...
  │     └── gap repair → list_transactions_since (return only; sync persists once)
  ├── enrich_lot_from_events + apply_child_orders_from_events
  ├── reconcile open / cancelled lots
  ├── validate_exposure (bootstrap + periodic, default 3600s)
  └── recompute instrument_exposure
```

**Source of truth:** OANDA trades → `broker_lots`. OANDA transactions → `broker_events`. OANDA positions are a **health check only** (aggregate sanity vs materialized exposure).

### Domain model

| Entity | MongoDB collection | Role |
|--------|-------------------|------|
| `PositionLot` | `broker_lots` | One OANDA trade (open/closed); embeds `stop_loss` / `take_profit` `ChildOrder` |
| `BrokerEvent` | `broker_events` | Immutable normalized transaction; cross-links `broker_lot_id`, `broker_order_id`, `batch_id` |
| `InstrumentExposure` | `instrument_exposure` | Rollup per `(exchange_id, account_id, symbol, direction)` |
| Sync cursor | `broker_sync_state` | `lastTransactionID`, bootstrap timestamp, exposure check time |

### Removed / deprecated

| Item | Replacement |
|------|-------------|
| `trades` repository / collection as primary ledger | `broker_lots` + `broker_events` |
| Parallel `_oanda_account_sync_loop` in web app | `run_broker_sync` only |
| `run_oanda_account_sync` bootstrap/poll logic | Thin wrapper → `run_broker_sync` |
| Legacy tick bots (`executor`, standalone `data_analyzer`, `brokers` sub-bots) | Secretary pipeline workers |
| `broker_sync.py` monolith | `trading/broker/sync.py` + adapters |

## Phase 1 — Sync consolidation

### Unified orchestrator

- All OANDA polling flows through [`run_broker_sync`](../../src/brokerai/trading/broker/sync.py)
- Web app runs a single trade sync loop (no lock contention with a second account loop)
- `run_oanda_account_sync` retained as a backward-compatible wrapper for secretary/settings callers

### Canonical bootstrap

[`run_oanda_bootstrap`](../../src/brokerai/trading/oanda_bootstrap.py):

1. `get_account_details` → open lots + `lastTransactionID`
2. `list_all_trades(state=CLOSED)` → closed history
3. Transaction idrange backfill → events + cursor; sets `account_bootstrap_at`

### Incremental sync

- One `/changes` poll per tick via `sync_incremental_from_changes`
- No `/openTrades` or `/positions` on every tick
- Live prices only when `fetch_live_prices=True` (broker monitor)

### Positions health check

- `validate_exposure` on bootstrap and at most once per `BROKERAI_OANDA_EXPOSURE_CHECK_INTERVAL_SECONDS` (default **3600**)
- Mismatches logged at WARNING; returned on `SyncResult.exposure_mismatches`; lots remain authoritative

### Cursor repair

- Shared [`oanda_cursor_repair.py`](../../src/brokerai/trading/oanda_cursor_repair.py) for stale cursor and transaction gaps
- Repair helpers **return events only**; `run_broker_sync` is the sole persistence point

### Foundation fixes

- `broker_lots` queries use canonical `open_time` / `close_time` fields (fixes closed-lot pagination)
- Extracted [`oanda_client.py`](../../src/brokerai/integrations/oanda_client.py) — rate limiting, keep-alive, shared HTTP

## Phase 2 — Semantic linkage & exposure

### Child-order state machine

[`child_orders.py`](../../src/brokerai/trading/broker/child_orders.py):

- `apply_child_orders_from_events` + `reconcile_child_order` wired after `enrich_lot_from_events`
- Trade snapshots primary; events validate `PENDING → FILLED/CANCELLED`
- `filling_event_id` / `cancelling_event_id` joined against `BrokerEvent` stream
- `infer_close_reason` uses closing txn types when embedded SL/TP is stale

### Order-level `/changes` ingestion

- `apply_account_changes` processes `ordersCreated`, `ordersFilled`, `ordersCancelled`
- Child-order patches merged into affected lots before upsert

### Materialized `InstrumentExposure`

- Collection `instrument_exposure`, key `(exchange_id, account_id, symbol, direction)`
- Recomputed idempotently at end of each `run_broker_sync`
- Consumed by `validate_exposure`, `GET /api/trades/exposure`, exchange dashboard panel

### API + UI

| Area | Change |
|------|--------|
| `GET /api/trades/exposure` | Per-instrument long/short rollups |
| Trade reasons | `stop_loss`, `take_profit`, `partial_close` in registry |
| Trade detail panel | Fill/cancel txn IDs on embedded child orders |
| Exchange dashboard | Open exposure summary panel |
| Trade candle chart | Windowed OHLCV, event flag overlay, detail overlay |

## Phase 3 — Performance, retention, multi-broker prep

### Bulk event upsert

- `BrokerEventsRepository.upsert_events_bulk` — `pymongo.bulk_write`, `UpdateOne` + `$setOnInsert` for `created_at`
- Batch size: `BROKERAI_BROKER_EVENTS_BULK_BATCH_SIZE` (default **500**)

### Transaction pagination

- `list_transactions_idrange` / `list_transactions_since` follow OANDA `pages[]` URLs (no 500-txn cap)
- `iter_transactions_idrange` / `iter_transactions_since` yield per-page batches

### Streaming bootstrap

- `run_oanda_bootstrap(event_sink=...)` bulk-upserts each page during step 3
- `run_broker_sync` always passes a sink; returns `events_streamed=True`, empty in-memory events
- Post-bootstrap enrichment reads from MongoDB via `list_events_for_lot`

### Tiered event retention

| Tier | Criteria | Retention |
|------|----------|-----------|
| Trade-linked | `broker_lot_id` / `broker_order_id` set, or trade-linked `event_type` | Indefinite |
| Low-value | No linkage + admin type (e.g. `MARGIN_CALL`, `DAILY_FINANCING`) | `retention_expires_at` = time + 90d |

- Partial TTL index on `retention_expires_at`
- Protected IDs from open lots and incomplete closes skip TTL
- Re-upsert `$unset`s `retention_expires_at` when an event becomes trade-linked or protected

### Multi-broker preparation (no second adapter yet)

- [`ids.py`](../../src/brokerai/trading/broker/ids.py) — `broker_lot_key`, `broker_event_key`, parse helpers
- `ExchangeConnectionsRepository.get_connection` / `save_connection` / `delete_connection`
- `BrokerStateService._credentials_for(exchange_id)`
- Lot lookups hardened to `(exchange_id, account_id, broker_lot_id)`

**Permanently deferred:** broker-native trailing-stop order churn; separate `child_orders` collection.

## Upgrade notes

1. **No mandatory migration script** — new collections are created on first sync. Existing deployments should run a **full broker sync** (or reconnect OANDA and allow bootstrap) to populate `broker_lots` and `broker_events`.
2. **Legacy `trades` collection** — no longer written by the application. UI and API routes read from `broker_lots`. Remove or archive legacy data manually if present.
3. **TTL index** — `retention_expires_at` TTL is passive; only low-value unlinked events expire. Trade-linked and protected events are never TTL'd.
4. **Restart services** after upgrade so a single sync loop is active and indexes are ensured.

## Known limitations (Alpha)

- **Forex execution only** — other asset classes remain stubs
- **Practice account recommended** for evaluation
- **OANDA closed trades** still require a separate API call; no single endpoint returns full open + closed history
- **Exposure mismatch** — logged and surfaced on sync result; no automatic repair
- **Trailing stops** — strategy-driven market closes only; no broker-native trailing order churn
- **Second broker** — ID helpers and connection CRUD exist; no non-OANDA adapter shipped
